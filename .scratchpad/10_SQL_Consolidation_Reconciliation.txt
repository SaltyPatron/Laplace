WHAT WAS WRONG
--------------
The extension shipped ONLY from the modular tree (sql/manifest.install -> cmake
SqlManifest.laplace_write_sql_chain -> #include chain -> cpp -> the built
laplace_substrate--<ver>.sql). Verified directly: laplace_substrate.sql.in is a
2-line shim; SqlManifest.cmake only emits manifest lines; no numbered file is
referenced by the build.

In parallel, 44 numbered NN_*.sql.in "migration-legacy" bundles still sat at
sql/ top level. They shipped nothing, but:
  * they perturbed EXT_VERSION via a GLOB_RECURSE sql/*.sql.in SHA-256 (footgun);
  * being tracked-but-never-`git rm`'d, any git checkout/reset resurrected them
    (the "keeps reverting/sabotaging" effect) — no generator regenerates them;
  * the migration was INCOMPLETE: real objects were stranded in numbered files
    and never made it into the manifest, so they silently were NOT shipping.

The incompleteness was proven, not assumed. Independent object-level inventory
found three stranded objects the "safe to delete" summaries missed:
  1. finish_consensus_fold_steps  (PROCEDURE, resumable partitioned fold) — only
     in 14_period_fold.sql.in; absent from the manifest; NOT shipping. Worse, the
     shipping functions/fold/consensus_fold_swap.sql.in references it in a comment
     as an existing capability.
  2. 43_table_tuning.sql.in (planner-stat + autovacuum tuning) — untracked,
     unique, in no manifest; NOT applied by any build.
  3. views/refresh_consensus_views.sql.in — a modular-tree file defining
     v_consensus_resolved + v_consensus_edges, in NO manifest (caught by the new
     completeness gate on its first run). Duplicate; bodies byte-equal to the
     shipping views/v_consensus_resolved.sql.in + views/v_consensus_edges.sql.in.

ZERO-LOSS PROOF (mechanical, reproducible)
------------------------------------------
Harness: extension/laplace_substrate/test/sql_parity/parity.py
Dollar-quote/comment-aware tokenizer. For EVERY object in EVERY numbered file it
asserts one of: IDENTICAL (normalized body == a modular body), CONFLICT
(reviewed, resolved to modular), FRAGMENT (numbered side truncated/invalid,
modular complete), MIGRATED (relocated verbatim; also asserts normalized-equal to
the numbered original), RETIRED (a modular DROP module exists). Anything else =>
UNACCOUNTED => exit 1.

Result on the pre-deletion tree:
  IDENTICAL (proven normalized-equal) ....... 122 objects
  CONFLICT (resolved to modular) ............   4
  FRAGMENT (numbered truncated) .............   5
  MIGRATED (finish_consensus_fold_steps) ....   1  (normalized-equal to original)
  RETIRED (content_descent_novel_ordinals) ..   1  (drop module present)
  EMPTY numbered files ......................  16
  UNACCOUNTED ...............................   0   => PASS

LOCKOUT (pattern made structurally impossible to reintroduce)
-------------------------------------------------------------
  * EXT_VERSION now hashes ONLY manifest-listed modules + shims + sqldefines
    (CMakeLists.txt), not a filesystem glob. A stray .sql.in can no longer perturb
    the version.
  * laplace_check_manifest_complete (cmake/SqlManifest.cmake) FAILS configuration
    if any sql/**/*.sql.in is neither manifest-referenced nor allow-listed (shims,
    sqldefines.h.in, sql/.manifest_ignore). GLOB uses CONFIGURE_DEPENDS so a newly
    added orphan triggers reconfigure + fails.
  * NEGATIVE TEST (real reconfigure, planted orphan):
      CMake Error at .../SqlManifest.cmake:83 (message):
        SQL manifest completeness check failed. ...
          .../functions/ops/44_i_am_an_orphan.sql.in
    Configure aborted before build. Orphan removed => configure passes again.
    This gate is what would have caught all three stranded objects above.
