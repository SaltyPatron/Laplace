SECTION 2 — THE ACTUAL MESS (what makes progress a battle)
================================================================================
2.1 INGESTION LANE FRAGMENTATION — the core finding. The pipeline core (1.3) is good,
    but there are AT LEAST SEVEN distinct ways a decomposer reaches it, and lane choice
    is historical accident, not policy (survey of all of Laplace.Decomposers, 2026-07-05):
      L1 raw SubstrateChangeBuilder hand-roll: Unicode, Code/Repo/Stack/Tabular/
         TinyCodes, Model, OpenSubtitles(main), FrameNetLuIngest, SemLink's four
         sub-ingest files (FnLuSynsetBridge/MapNet/PredicateMatrix/RoleMapping)
      L2 DecomposerBatch: ISO, FrameNet, PropBank, VerbNet, WordNet
      L3 StructuredGrammarIngest: Atomic2020, ConceptNet, OMW, Tatoeba, Wiktionary
      L4 NativeGrammarIngest + EtlWitness/EtlManifest ETL lane: Atomic2020, ConceptNet,
         OMW, Wiktionary — i.e. FOUR SOURCES HAVE TWO PARALLEL IMPLEMENTATIONS (L3+L4).
         CLI dispatch (IngestCommands.cs:131-147) resolves the ambiguity by special-
         casing wiktionary/omw to their L3 decomposers BEFORE the EtlManifest.IsRoutable
         check, while atomic2020/conceptnet fall through to EtlDecomposer. The losing
         lane in each pair is live code that looks authoritative and isn't.
      L5 IngestBatchPipeline direct: UD, SemLinkIngestAdapter
      L6 ContentWitnessBatch: WordNet, OpenSubtitlesZipIngest, UdIngestAdapter,
         SemLinkIngestAdapter (overlaps L2/L5 — a second axis of divergence)
      L7 PCoreParallelCompose: CILI (and the chess lane in Laplace.Chess.Service)
    Consequences: Rule #8 conformance depends on which lane a source happens to sit in;
    every new source is written by copying the nearest neighbor, forking further; a
    pipeline fix (like Issue 44's run bracket) lands in one lane and must be re-verified
    in seven. This is the "childish school project" smell — not bad code per file, but
    N parallel half-standards.

SECTION 3 — TARGET STATE (what "stable clean" means, concretely)
================================================================================
T1 ONE ingestion spine. A decomposer is a pure function: container bytes → stream of
   generic records. It knows nothing about batching, probing, dedup, Glicko, COPY, or
   index cycling. The spine (IngestBatchPipeline working-set mode →
   ConsensusAccumulatingWriter → NpgsqlWorkingSetApply, bracketed by IngestRunner) is
   the ONLY door to the database. Adapters become thin, declarative configs over the
   spine, not parallel implementations.
T2 One door per read, too: every logical read op has exactly one substrate entry point
   (07 §4.2's list) and both SubstrateClient and QueryCommands consume the same shared
   surface.
T3 Structure tells the truth: Substrate project folders separate contract/spine/
   adapters/source-support; the losing ETL-vs-Structured lane is deleted, not dormant.
T4 Gates make regressions structurally impossible (the doc-10 pattern, generalized):
   an architecture test fails the build if production decomposer code references the
   writer/apply layer directly or contains inline SQL; the SQL manifest gate already
   covers the extension side.
T5 Docs stay load-bearing: 02 compacted to current-truth; 06/07/CLAUDE.md re-baselined;
   one place per fact.
T6 The foundry build (09/12 gap list) proceeds on top of a spine that no longer forks
   under it — architecture read off the data, routing fidelity at depth measured.
