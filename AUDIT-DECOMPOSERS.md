### [CRITICAL/CONFIRMED] accuracy-correctness — Root-scoped JSON property lookups scan the WHOLE AST and take the FIRST matching key, not the top-level object's field
- **Where:** `app/Laplace.Decomposers.Abstractions/JsonGrammarHelper.cs:185`  ·  src `witness-correctness-A`
- **Detail:** TryPropertyStringSpan (l.185), TryArrayPropertyNode (l.214), and ObjectNodesInArrayProperty (l.48) iterate every node in the flattened AST and return the FIRST 'pair' whose key matches, with NO constraint that the pair is a direct child of the root object. WiktionaryGrammarWitness uses these for 'word', 'lang_code', 'lang', 'pos', 'etymology_text' (root word identity) and for the root-level RelMap walk (WalkRootRelations, l.152-159). In wiktextract a sense object contains its own {word:...} relation entries and nested arrays named 'synonyms'/'antonyms'/etc. Because the scan is pre-order and unscoped, ANY input where a nested same-named key occurs before the root's (or where the root lacks the key but a sense has it) yields the wrong value: e.g. WalkRootRelations will pull a SENSE's 'synonyms' array when the root object has none, emitting those edges a SECOND time at root scope with context=null (duplicate + misattributed), and a root 'word' could resolve to a relation target word inside a sense if 'senses' is serialized before 'word'. This produces WRONG subject identity for the entire record and spurious/duplicate relation edges.
- **Fix:** Add a root-object-scoped overload: resolve the root object node once (JsonGrammarHelper.FindRootObjectNode already exists) and route TryComposedProperty/TryPropertyUtf8/ObjectNodesInArrayProperty through TryObjectPropertyStringSpan/TryObjectArrayPropertyNode with that root index, exactly as the per-sense helpers already do. Pass the root object node index into GrammarComposeContext (add an int RootNodeIndex) so witnesses never do an unscoped key scan.

### [CRITICAL/CONFIRMED] accuracy-correctness — JSON string-leaf entity IDs (compose path) are computed by a different algorithm than ContentWitnessBatch, so Wiktionary edges don't link to WordNet/OMW/VerbNet entities for the same surface
- **Where:** `engine/core/src/grammar_compose.cpp:266`  ·  src `witness-correctness-A`
- **Detail:** compose composes a JSON string value as a SINGLE tier-2 node built directly from ALL of the string's grapheme children (json_leaf_fill_grapheme_children, tier hardcoded to 2 at l.272; id = hash128_merkle(2, [all grapheme ids]) via hash_composer_compose_node). WiktionaryGrammarWitness resolves every relation target / word subject through GrammarRowComposer.TrySpanEntity, i.e. these compose-path IDs. WordNet, OMW and VerbNet instead emit content via ContentWitnessBatch -> TextEntityBuilder.TryDecomposeRoot -> text_decomposer_run, which builds the full codepoint->grapheme->word->sentence hierarchy and takes natural_unit_index/collapse_idx as the root. For any multi-word/multi-grapheme surface (definitions, examples, phrasal lemmas like 'New York', multi-word synonyms/coordinate terms) the two roots are structurally different merkles and produce DIFFERENT Hash128 ids. Result: a Wiktionary 'café'/'New York'/gloss node is a separate substrate entity from the identical WordNet/OMW node, so cross-source edges silently fail to converge and the graph fragments.
- **Fix:** Make the JSON string-leaf composition go through the same content tier-tree path as ContentWitnessBatch (call laplace_text_decomposer_run + natural_unit_index on the decoded string bytes and use that root id for the span entity), or have the witness resolve relation targets/words via ContentWitnessBatch.RootId(decoded utf8) instead of TrySpanEntity. Whichever direction, the JSON path and the content path MUST share one root-id function.

### [CRITICAL/CONFIRMED] accuracy-correctness — LoadTensorF32 trusts caller's expectedElements with no validation against the actual tensor byte length; mismatched config.json silently corrupts weights via heap over-read.
- **Where:** `app/Laplace.Decomposers.Model/WeightTensorETL.cs:37`  ·  src `decomposer-correctness-C`
- **Detail:** result=new float[expectedElements] and the decode loops read s[i] for i in [0,n) where n=expectedElements, but raw is only tref.DataLength bytes and nothing checks raw.Length >= expectedElements*sizeof(dtype). The single caller ModelTokenEdgeETL.cs:75 passes (long)vocab*d derived from config.json (vocab_size x hidden_size). When config.json's vocab_size/hidden_size do not match the embedding tensor's real shape (vocab padding to a multiple of 64/128 is extremely common, and config vs safetensors drift happens on re-quantized/edited checkpoints), the loop walks past the end of raw[] reading adjacent heap memory. ### [CRITICAL/CONFIRMED] perf-altitude — Streaming row path parses each line through laplace_grammar_parse -> ts_parser_new PER LINE; a million-line JSONL file allocates a million parsers
- **Where:** `engine/core/src/grammar_rows.c:59`  ·  src `grammar-spine-perf`
- **Detail:** laplace_grammar_row_iter_feed splits the carry buffer on '\n' and at line 59 calls laplace_grammar_parse(it->carry+start, row_len+1, it->recipe, &ast) for every non-empty line. ### [CRITICAL/CONFIRMED] perf-marshalling — GrammarAst.NodeTypeName does a P/Invoke (laplace_ast_type_name) plus Marshal.PtrToStringUTF8 string allocation on EVERY call; it is invoked once per node per scan, and the witness walk does dozens of full-AST scans per row.
- **Where:** `app/Laplace.Engine.Core/GrammarDecomposer.cs:71`  ·  src `managed-marshalling-perf`
- **Detail:** NodeTypeName(uint) crosses the boundary via NativeInterop.AstTypeName then allocates a fresh managed string with Marshal.PtrToStringUTF8 (line 75) for every call. The native side (engine/core/src/grammar_decomposer.c:103-106 laplace_ast_type_name) merely returns ts_language_symbol_name(lang, type_id) — a pointer into tree-sitter's static symbol table, no allocation. JsonGrammarHelper calls NodeTypeName inside O(N) loops at lines 55,68,81,90,104,120,123,132,175,179,191,207,219,223,238,242,269,281,298,311 — and those loops are re-run for every property lookup. This removes both the per-node crossing and the per-node string allocation.

### [CRITICAL/CONFIRMED] perf-parallelism — Row composition is single-threaded in every commit-policy path: only the DB commit fans out, never the decompose/parse/materialize work
- **Where:** `app/Laplace.Ingestion/IngestRunner.cs:164`  ·  src `decomposer-parallelism`
- **Detail:** All three execution paths (StrictSerial lines 164-187, Unordered lines 301-316, EpochBarrier lines 378-393) run the decomposer's IAsyncEnumerable in exactly one producer `Task.Run(async () => await foreach (var intent in decomposer.DecomposeAsync(...)))`. For CODE and STRUCTURED/STREAMING decomposers, DecomposeAsync is where the cost lives: native tree-sitter parse, GrammarRowComposer.Materialize (P/Invoke marshalling per entity/phys/precedes), and IGrammarWitness.WalkRow (O(N^2-3) JsonGrammarHelper scans). The Unordered/EpochBarrier paths spin up `options.ParallelWorkers` CONSUMER tasks, but those only call `_writer.ApplyManyAsync` — they parallelize the DB COPY/merge, not the CPU-bound composition. Trigger: a multi-million-row JSONL (Wiktionary/ConceptNet) or a large code tree pins one core at 100% on parse+compose while N-1 cores and the DB sit idle. The producer cannot keep the bounded channel full, so the parallel consumers starve and the whole `ParallelWorkers` knob delivers near-zero speedup on producer-bound sources.
- **Fix:** Introduce a parallel composition stage between DecomposeAsync's row iterator and the commit channel. For STRUCTURED: have StructuredGrammarIngest emit raw (lineUtf8, nativeAstHandle) items into a bounded Channel, then run `ParallelWorkers` worker tasks (or Parallel.ForEachAsync over the channel) that each own a GrammarRowComposer + per-worker SubstrateChangeBuilder, Materialize+WalkRow independently, and write finished SubstrateChange to the commit channel. Composition of independent rows is embarrassingly parallel and order-independent at the per-row entity/physicality level (consensus folds commutatively). Keep epoch ordering by tagging each produced change with its source CommitEpoch and letting the existing EpochBarrier flush gate on it. This is the single highest-leverage change.

### [CRITICAL/CONFIRMED] perf-parallelism — Global mutable dedup banks (g_canon_slots / g_entity_slots) with zero synchronization make the content-witness path unsafe to parallelize
- **Where:** `engine/core/src/content_witness_batch.c:213`  ·  src `decomposer-parallelism`
- **Detail:** The content-witness dedup state is module-global: g_canon_slots/g_canon_cap/g_canon_count (lines 213-215) and g_entity_slots/g_entity_cap/g_entity_count (lines 217-219). canon_lookup/canon_insert (lines 223-267) and entity_record/entity_contains (lines 269-310) read and mutate these without any lock or atomics, and content_witness_batch_add (line 383) drives them per content unit. IntentStage.ResetContentBank is called once per run (IngestRunner.cs:85). If row composition is parallelized (the fix above), two worker threads calling content_witness_batch_add concurrently will race on the open-addressing tables — including realloc/free during rehash (lines 242-256, 275-289) — causing torn reads, double-frees, and lost/duplicated dedup decisions (wrong substrate: missing or duplicated content entities). - **Fix:** Make the bank per-worker or thread-safe. Cleanest: replace the file-global statics with a caller-owned `content_witness_bank_t*` handle passed into content_witness_batch_add, so each composition worker holds its own bank (dedup is a perf optimization, not a global invariant — per-worker banks at worst re-emit a few duplicate entities that the writer's idempotent merge collapses). If a shared bank is required, guard canon_*/entity_* with a rwlock and make the hash tables grow under the write lock, or shard the table by id.lo and lock per shard.

### [HIGH/CONFIRMED] accuracy-correctness — Surface strings are taken as RAW JSON bytes with no escape decoding, so any escaped value gets a wrong/non-convergent entity id
- **Where:** `app/Laplace.Decomposers.Abstractions/JsonGrammarHelper.cs:265`  ·  src `witness-correctness-A`
- **Detail:** TryStringNodeSpan returns the byte span of the 'string_content' node verbatim, and TryComposedNode/TrySpanEntity hash those raw bytes. JSON escapes are never decoded: a value stored as é, \", \\, \/, \n, \t in the source line is composed from the literal backslash-escape bytes. So 'café' composes the grapheme sequence c,a,f,\,u,0,0,e,9 instead of c,a,f,é. The entity id therefore (a) does not match the decoded surface emitted by every other decomposer (which decodes via Encoding.UTF8.GetString/GetBytes), and (b) is semantically wrong (it indexes the escape text). wiktextract uses \u escapes for non-Latin scripts and IPA, and \" inside example sentences, so a large fraction of non-ASCII Wiktionary surfaces are affected. Utf8ToString (l.154) used for lang_code/pos/tag/etymology-name comparisons has the same gap but those fields are ASCII so only the entity-id path matters.
- **Fix:** Before composing/looking up a string value, JSON-unescape the string_content bytes (handle \", \\, \/, \b,\f,\n,\r,\t and \uXXXX surrogate pairs) into a decoded UTF-8 buffer, then resolve the entity from the decoded bytes via the shared content root-id path. Equivalently, route string values through System.Text.Json's reader-decoded GetString() as the obsolete WiktionaryWitness already does.
