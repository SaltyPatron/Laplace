ISSUE 10 — seed-step.cmd "framenet" branch used an anomalous inline parenthesized IF block
Status: FIXED this session
Every other step dispatches via `if /i "%STEP%"=="x" goto run_ingest`. framenet alone used

ISSUE 11 — env.cmd has no idempotency guard; PATH grows unboundedly across repeated calls
Status: FIXED this session, verified
env.cmd unconditionally prepends 5 PATH segments (plus re-runs Intel oneAPI's own
setvars.bat, which does the same) every time it is `call`ed, with no guard against being
sourced more than once in the same process. seed-foundation.cmd — a NEW file, added in the
same commit range as the other seed-* scripts — calls seed-step.cmd (which calls env.cmd)
once per foundation source, all inside one cmd.exe process. Measured: PATH grew ~603
characters per call (4615 -> 5218 -> 5821 -> 6424...), which by the 5th-6th step in a 10-step
loop pushes past cmd.exe's ~8191-character line-length ceiling, producing "The input line is
too long" and corrupted/truncated command execution. No caller before seed-foundation.cmd
ever invoked env.cmd more than once per process, so this was always latent and never
triggered until now. Fixed by adding to the top of env.cmd:
    if defined LAPLACE_ENV_LOADED exit /b 0
    set "LAPLACE_ENV_LOADED=1"
Verified: PATH length now holds flat (4615) across 4 repeated calls in one process instead of
growing. This is a real, load-bearing fix — distinct from Issue 12 below, which is a separate
bug that was actually responsible for the specific 'ocal' symptom chased for most of this
session.

ISSUE 12 — 'ocal' is not recognized (ROOT CAUSE: PowerShell 7 / Windows 11 regression, not
a Laplace bug)
Status: RESOLVED — root cause identified, not fixable in this repo, workaround adopted
For most of this session, invoking any .cmd file in scripts/win/ via the PowerShell tool
(cmd /c "...", direct invocation, `$var; cmd /c $var`, all styles tried) produced:
    'ocal' is not recognized as an internal or external command, operable program or batch
    file.
immediately, before any script output, including with zero arguments. Extensive bisection
wrongly suspected file corruption (raw bytes checked clean via xxd), PATH growth (real bug,
Issue 11, but not the cause of this specific symptom — it persisted even in a single fresh
process with no accumulation possible), and the framenet parenthesized block (real anomaly,
Issue 10, but fixing it did not resolve 'ocal'). Actual root cause, confirmed via web search:
GitHub issue PowerShell/PowerShell#27634 — "PowerShell & operator fails to invoke .cmd/.bat
files ... (Windows 11 KB5095093 regression)". After that Windows update, PowerShell 7's
process-creation logic for launching .cmd/.bat files mis-slices the constructed command
line. Every .cmd in this repo has `setlocal EnableDelayedExpansion` as line 2 — a
truncated/mis-sliced launch landing partway into that line reads exactly as
"'ocal' is not recognized" (a fragment of "setlocal"). Confirmed by elimination: the Bash
tool (git-bash/MSYS2, a completely different process-spawning code path, not pwsh's
Process/CreateProcess route) never reproduced the symptom under any invocation style tested.
Workaround adopted for the rest of this session and recommended going forward on this
machine: invoke all scripts/win/*.cmd files through the Bash tool (`cmd //c "scripts\\win\\
x.cmd args"`), never through the PowerShell tool directly. This is an OS/PowerShell defect on
this Windows 11 build, not something to "fix" in the repo.
Minor, harmless side-effect noticed once routed through the Bash tool: the
`tasklist ... | find /I "Laplace.Cli.exe"` mutex-guard and wait-for-exit poll in
seed-step.cmd's :run_ingest_impl/:wait_cli_exit resolve `find` to git-bash's
/usr/bin/find.exe (GNU findutils) instead of Windows' System32\find.exe, since MSYS2's PATH
puts its own /usr/bin ahead of System32 for children spawned that way. This produces
cosmetic "find: '/I': No such file or directory" / "find: 'Laplace.Cli.exe': No such file or
directory" lines but does not block ingestion — real ingest work completes successfully
around it (verified: iso639 ingest completed fully — 43,088 relations materialized,
layer_complete=True — despite these lines appearing at the tail). Not fixed; flagging as a
known cosmetic side-effect of the Bash-tool workaround, not worth chasing further unless it
starts actually blocking something.

ISSUE 15 — concurrent ingest crash (real, not hypothetical)
Status: OPEN — needs a decision (guard concurrent ingests properly, or confirm it's safe and
fix the crash)
Ran OMW and a document ingest concurrently against the same database as a deliberate test
("can we run two ingests at the same time? Let's find out"). Both progressed for several
minutes, then the document ingest hit:
    PostgresException 23505: duplicate key value violates unique constraint "entities_pkey"
    Key (id)=(\x533fc56983559b11986ef008b897a2b1) already exists.
followed immediately by an unhandled ObjectDisposedException on IntentStage
(IntentStage.get_EntityCount, via NpgsqlSubstrateWriter.ApplyManyAsync ->
ConsensusAccumulatingWriter), which aborted the whole run rather than being caught/retried.
The seed-step.cmd mutex guard (see Issue 16) does not prevent this scenario — it does not
detect the already-running process, so nothing stops two ingest processes from racing on the
same content-addressed insert path. Two possible fixes worth deciding between: (a) make the
duplicate-key case a proper retry/skip path instead of a fatal abort (content-addressing
means a duplicate key here should mean "someone else already wrote this exact content,"
which is not actually an error), or (b) make the mutex guard actually work so concurrent
ingests can't happen at all. (a) seems more in the spirit of the content-addressing design;
(b) is the more conservative fix.

ISSUE 17 — apply_batch.c had no protection against concurrent content-addressed writes
Status: FIXED this session, verified against the original repro
Root cause of the entities_pkey/physicalities_pkey unique-violation crash first hit when OMW
and a document ingest ran concurrently (see the original incident above). The anti-join dedup
(LEFT JOIN ... WHERE id IS NULL, extension/laplace_substrate/src/apply_batch.c) is correct in
shape but has a check-then-act race: two concurrent apply_batch calls can both see the same
content-addressed id as "not yet present" before either commits, both attempt the insert, one
wins, one hits a raw unique-violation. This is not a normal application error to retry around
-- content-addressing means the "conflict" is semantically "we agree," not "something went
wrong" -- so the fix is prevention, not a retry/ON CONFLICT reaction. Added
`SELECT pg_advisory_xact_lock(hashtextextended('laplace_apply_batch', 0))` immediately after
SPI_connect() in laplace_apply_batch, serializing concurrent apply_batch calls against each
other (transaction-scoped, auto-released at commit/rollback, acquired once per already-bulk
batch call, not per row). Rebuilt, redeployed, reseeded from a fresh database afterward;
unicode/iso639/cili/wordnet/verbnet/propbank all completed cleanly under the new build with
no recurrence of the race.
Note: TransientErrorRetryPolicy.ConcurrencyRetry (app/Laplace.Ingestion/
TransientErrorRetryPolicy.cs:32-37) already correctly classifies 23505/40001/40P01 as
transient and was firing correctly (confirmed via log: "Transient ingest error... will retry
whole batch") -- the retry classification was never the bug. What made the retry itself fail
was NpgsqlSubstrateWriter.ApplyManyAsync unconditionally disposing the caller-owned
IntentStage objects (prebuiltStages, i.e. batch's own SubstrateChange.IntentStages) in its
finally/else branches on every call including a failed one about to be retried by the same
batch -- an ObjectDisposedException on IntentStage.get_EntityCount masking the real duplicate-
key issue. This disposal-ownership bug was found but NOT fixed/kept -- superseded by the
prevention-based fix (the advisory lock), since preventing the race outright removes the need
for that retry path to ever fire for this specific cause. If 23505/40001/40P01 legitimately
recur from some other cause, the disposal-ownership bug is still latent and would still need
fixing separately.

ISSUE 19 — descent_probe.c: O(n^2) traversal in laplace_content_descent_bitmap_core
Status: FIXED this session, not yet verified against a real ingest (rebuild/redeploy pending)
extension/laplace_substrate/src/descent_probe.c: the DFS/stack-based descent that clears
bitmap bits for "not yet present" nodes rescanned the ENTIRE n-length parents[] array to find
each popped node's children (`for (i = 0; i < n; i++) if (parents[i] == node) ...`), inside
the while-loop that visits every node once -- O(n^2) total instead of O(n). For a large
document (tens of thousands of codepoints/graphemes/words per probe call), this is hundreds
of millions of comparisons per call and gets worse as ingested files get bigger, which matches
what was observed live: a single content_descent_bitmap call actively running 53+ seconds
(confirmed via pg_stat_activity: state=active, no wait_event, so genuinely computing, not
blocked) partway through a real document ingest run. Fixed by building a children-index once
(bucketed linked-list by parent, O(n) construction) before the traversal instead of rescanning
per node. IMPORTANT CAVEAT (author's correction, preserved verbatim in spirit): this fixes the
within-call algorithmic complexity only. It does NOT address the deeper, repeatedly-stated
architectural point of this session -- that descent probing (and dedup generally) should be a
genuine BULK, tier-scoped operation across the whole in-flight batch/corpus, not a per-
document call no matter how efficient each individual call's internal algorithm is. Each of
the 34 files in a document ingest still gets its own separate probe call and SPI round-trip
today. The O(n)-per-call fix is real and worth keeping, but it is explicitly NOT the O(tier)
bulk-dedup architecture the author is asking for -- flagged as still-open, larger scope.

ISSUE 25 — FIXED, verified live, corpus-wide.
Root cause (confirmed via a dedicated multi-agent root-cause pass, not guessed):
app/Laplace.SubstrateCRUD/TierTreeDescent.cs's ProbeBatchEmitBitmapsAsync called
reader.MarkProven(ids) unconditionally for every id in a probe batch, BEFORE checking which
ids the descent bitmap actually confirmed present -- poisoning NpgsqlSubstrateReader's
process-lifetime `_proven` cache with ids that were genuinely absent. The next lines in the
same function then re-check `_proven` first, see the false "proven" entry, and report those
ids as present -- overwriting the correct "absent" result from the real DB probe two lines
earlier. This flows into merkle_dedup_trunk_shortcircuit (native), which skips emitting the
entity. Since `_proven` never clears, this is a one-time poisoning per unique id -- explains
why a single id (a `\r` grapheme, not "Dorian") was referenced 1,622 times in dorian.txt's
trajectory: it's the most common recurring character in a Windows-line-ended text file, and
its first-ever probe permanently poisoned the cache for the rest of the run.
Fix, comprehensive, all verified live/build-checked:
  - TierTreeDescent.cs: MarkProven gating fixed (root fix).
  - descent_probe.c: replaced the "assume 0xFF present, clear-on-disproof" scheme with a
    positive-confirmation-only tier_batch_existence_probe primitive (new SQL/native function),
    applied uniformly to all tiers including tier-0 (no special-casing).
  - hash128.c: hash128_merkle now mixes `tier` into the hash instead of discarding it (was
