architecture), 02_Identified_Issues.txt (whole-repo live-execution issue tracker, same
statuses/conventions reused here: OPEN / FIXED / CLOSED-NOT-A-BUG / NEEDS-VERIFICATION).

Why chess matters here, restated from CLAUDE.md: `attestations.outcome CHECK IN (0,1,2)` and
this repo's `PlyOutcome {Loss=0,Draw=1,Win=2}` are bit-for-bit the same encoding on purpose,
and Glicko-2 (literal chess rating math) scores epistemic confidence for every fact in the
substrate. Chess is the one domain where the fold's convergence to ground truth is objectively
checkable — which makes bugs in chess's own evidence pipeline unusually consequential: they
don't just mis-score a chess fact, they're evidence the whole consensus mechanism can be fooled.

ISSUE C01 — cutechess-cli's real self-play PGN format is silently unparseable by this
codebase's own eval/clock extractors, for exactly the ground-truth verification loop CLAUDE.md
calls out as chess's whole reason for architectural importance
Status: OPEN — SELF-VERIFIED against a real generated PGN, not inferred from reading tests
`app/Laplace.Chess.Service/PgnEvals.cs:8` and `PgnClocks.cs:7` both hardcode a regex that ONLY
matches the lichess-standard bracketed annotation format: `\[%eval\s+([^\]]+)\]` and
`\[%clk\s+(\d+):(\d+):(\d+(?:\.\d+)?)\]`. Every test fixture for these two files
(`PgnGamesTests.cs:144,182`, `PgnClocksTests.cs:8-10`) uses that exact bracketed format, so the
test suite never exercises anything else. I built the real vendored cutechess-cli
(scripts/win/build-cutechess.cmd, unmodified, first try) against the user's real Qt 6.8.3 + real
Stockfish 18 binary and ran an actual 2-game match with `-pgnout`. The real output comments look
like `{+0.48/17 0.13s}` (eval/depth/time-spent) — no `[%eval ...]` wrapper, no `[%clk ...]`
wrapper at all, ever (confirmed by grepping the entire cutechess C++ source tree for `%eval`/
`%clk`: zero matches — it never emits that format under any settings). `CutechessRunner.cs:61`
invokes the exact same cutechess-cli binary with `-pgnout`, so any local self-play tournament run
through this repo's own lab layer produces PGN in this bare format. `ChessPgnDecomposer.cs:77-90`
feeds that PGN straight into `PgnClocks.ClockTokens`/`PgnEvals.EvalTokens`, both of which will
match zero times and return null for every such game — per-move engine-strength evaluation and
clock/time-pressure signal (`HAS_EVAL`, `HAS_CLOCK`, `HAS_EVAL_TOKEN`, `HAS_THINK_CLASS`,
`MOVE_QUALITY`'s think-factor input) is silently dropped for the entire local cutechess-produced
corpus. Only Lichess-fetched games (which genuinely export in `[%eval]`/`[%clk]` format) get this
signal. Given Issue C03 below (the lab layer has no automated ingestion path at all — a human
must manually run `laplace ingest chess <path>` on lab output), this bug would fire silently on
literally every real invocation of the one automated path that exists.
Fix shape (not yet done): add a second regex branch recognizing cutechess-cli's bare
`{[+-]?\d+\.\d+/\d+ [\d.]+s}` comment format, or normalize cutechess's PGN comments to the
lichess bracketed convention at the point CutechessRunner writes them out.

ISSUE C02 — chess's entire relation-type vocabulary (~25 types) is absent from
engine/manifest/relation_types.toml, which structurally guarantees highway_mask is permanently
zero for every chess attestation, forever — not a propagation bug, a registration gap
Status: OPEN — SELF-VERIFIED by direct grep, independently confirmed by a second agent's
separate trace through HighwayPerfcache.MaskForRelationType
Grepped `engine/manifest/relation_types.toml` (1252 lines) for all of MOVE, OUTCOME, PLAYED_BY,
HAS_RATING, OPENING_NAME, HAS_ECO, HAS_WHITE, HAS_BLACK, HAS_EVENT, ON_DATE,
HAS_TIME_CONTROL, HAS_TC_CLASS, HAS_TERMINATION, HAS_RESULT, GAME_AT, GAME_AT_PLY, HAS_EVAL,
MOVE_QUALITY, HAS_CLOCK, HAS_EVAL_TOKEN, HAS_THINK_CLASS, GAME_HAS_OPENING, GAME_HAS_ECO,
GAME_HAS_MOTIF, EXPLAINS, IS_EXAMPLE_OF — zero matches for every one of them (only "DEFINES"
incidentally exists, generically, not chess-specific). These ~25 relation types are instead
minted purely as `EntityTypeRegistry.Id("...")` (ChessVocabulary.cs:15-83) — a raw
`Hash128.Blake3(name)` (HighwayPerfcache.NodeHash, HighwayPerfcache.cs:7-20) with no
corresponding entry ever generated into `laplace_highway_perfcache.bin`. Traced the consequence
precisely: `SubstrateChangeBuilder.AddAttestation` calls
`HighwayPerfcache.MaskForRelationType(row.TypeId)` (HighwayPerfcache.cs:100-111), which returns
`Mask256.Zero` whenever the native `HighwayTableRelationByHash` lookup misses (rc != 1) — and it
will always miss for these hashes, by construction, since the perfcache is compiled from the
manifest file that never mentions them. This is a stronger, more precisely-diagnosed instance of
02_Identified_Issues.txt's Issue 01 (highway_mask NULL on ISO639 attestations, cause left open
there as "either the code path never calls it, or the perfcache wasn't loaded") — for chess,
the cause is not ambiguous: there is no entry to find, period, regardless of ingestion code
correctness or perfcache load timing. Same root mechanism likely explains Issue 01 too, worth
checking whether ISO639's relation types are similarly absent from the manifest.
Secondary consequence, not yet independently verified: `EntityTypeRegistry.cs` mints ~30 other
non-chess relation/entity type names the same way (Codepoint, Grapheme, Word, WordNetSynset,
VerbNetClass, etc.), so this "raw-hash mint bypasses the curated manifest" pattern is likely
repo-wide, not a chess-specific oversight — chess is simply where this session happened to
trace it to ground.
Open question: does highway_mask matter for CORRECTNESS of chess recall/search (would a chess
fact fail to be found at all) or only for acceleration (an optimization the manifest's own SQL
comments suggest it's meant for)? Not verified this session — flagged, not answered.

ISSUE C03 — the chess "lab" (cutechess tournaments, Lichess bot) has no automated path into the
substrate at all; the one ingestion endpoint that claims to do this is a stub that misreports
its own effect
Status: OPEN
`EndpointMappings.Chess.cs`'s `/chess/lab/jobs/{jobId}/ingest` endpoint returns
`{ queued = true, path, hint = "laplace ingest chess \"{path}\"" }` — it performs zero ingestion;
it only checks the artifact file exists and hands back a CLI command string for a HUMAN to run
manually. `queued: true` is actively misleading — nothing is queued anywhere. Separately,
`LichessBot.PlayGameAsync` (LichessBot.cs:120-181) streams a game to completion and logs
"game finished" but contains no code anywhere that records the final outcome, let alone attests
it — unlike `ChessEngineService.RunSelfPlayAsync`/`RunStrongSelfPlayAsync`, which do call
`_host.LearnGameAsync` to write real self-play results via `ConsensusAccumulatingWriter`. That
self-play writer path is invoked only from `Laplace.Cli/ChessCommands.cs`, a completely separate
call path from `ChessLabService`/the lab HTTP API. Net effect: cutechess tournament results and
lichess bot games currently have no automatic route into `attestations`/`consensus` at all —
only CLI-driven self-play does. Combined with Issue C01, even the manual `laplace ingest chess`
escape hatch silently drops eval/clock signal for cutechess-sourced PGN specifically.

ISSUE C04 — chess HTTP API surface is entirely unauthenticated, unlike the sibling billing
endpoints in the same file family, and combines with a real path-traversal write primitive
Status: OPEN (security)
Every route in `EndpointMappings.Chess.cs` (`/chess/new`, `/chess/legal`, `/chess/move`,
`/chess/bestmove`, `/chess/train/start|stop`, all `/chess/lab/*`) has zero
`.RequireAuthorization()`/tenant-resolution, whereas every route in the sibling
`EndpointMappings.Billing.cs` in the same assembly explicitly threads an `ITenantResolver`
through. `Program.cs` registers no global auth middleware. Concretely reachable consequences:
an unauthenticated caller can (a) start unbounded background self-play/train jobs (CPU
exhaustion), (b) launch arbitrary cutechess-cli tournaments (external process spawn), and (c)
exploit `ChessLabRunners.cs:155-158`'s `RunLichessFetchAsync`, which builds
`Path.Combine(LabDir, jobId, $"{user}_{site}.pgn")` from an unsanitized `user` value taken
directly from the job's public `Config` dict (`/chess/lab/start` request body) — unlike
`ChessGameFetcher.DefaultOut`'s own `Sanitize()` (alnum/-/_ only), this call path never
sanitizes `user`, and `Path.Combine` will honor a rooted/`..`-laden value, giving an
unauthenticated caller an arbitrary-file-write primitive on top of zero auth.
Fix shape: gate `/chess/*` behind the same auth mechanism as `/v1/billing/*`; reuse
`ChessGameFetcher.Sanitize()` (or equivalent) on `user`/`site` before path construction in
`ChessLabRunners.cs`.

ISSUE C06 — CountOutposts (feature-token extraction) has two real correctness bugs, plus a
matching test-design gap that explains why they shipped unnoticed
Status: OPEN (correctness bug), scope-limited to the `LAPLACE_CHESS_REKEY=1` opt-in feature-token
path (not the default position-identity/repetition path)
`PositionContent.cs:100`: `bb.Of(white ? Piece.WKnight : Piece.BKnight) | bb.Of(white ?
Piece.WBishop : Piece.WBishop)` — the bishop half of the ternary has both branches identical
(`Piece.WBishop` regardless of color), so Black's outpost-candidate computation ORs in White's
bishop bitboard instead of Black's. `PositionContent.cs:108-109`: the "is this outpost square
defended by a pawn" check tests `RankMask(r)` (the SAME rank as the piece) instead of the rank
BEHIND it — pawns never defend same-rank squares, so this branch is effectively always false for
genuine pawn support. `PositionContent.cs:107` has a third, related dead ternary (both branches
byte-identical) suggesting the color-conditional logic here was never finished. Why this shipped:
`PositionContentTests.cs:70-87` (`Surface_Rekey_AddsFeatureTokens_WhenEnabled`) only asserts the
`outpost:`/`mob:`/`open:`/`kzone:` TOKENS ARE PRESENT in the output string, never that their
VALUES are correct — the exact shape of test gap that would hide these three bugs.

ISSUE C07 — self-play player-name claims get the same trust weight as curated academic corpora
(hardcoded, not source-conditional)
Status: OPEN (correctness bug / consensus-poisoning vector)
`ChessVocabulary.EmitPlayer` (ChessVocabulary.cs:94-101) hardcodes
`SourceTrust.AcademicCurated` (0.85) for every `HAS_NAME_ALIAS` attestation it writes,
regardless of the actual `sourceId`/trust-class its caller is otherwise using. This is correct
for `ChessPgnDecomposer.EmitPlayer` (real PGN corpora, AcademicCurated is the right trust class)
but the SAME helper is also called from `SubstrateTurnHost.LearnGameAsync` for self-play games,
whose own `SelfPlayTrustClass = "ResponseContent"` implies a much lower trust weight (~0.20 per
`WitnessConstants.cs:37`) — so the player-name claim ("this game was played by 'Laplace'") is
attested at curated-corpus confidence even for self-play, silently decoupled from the trust
class the rest of that same code path correctly uses for every other attestation in the game.

ISSUE C08 — move-quality judgment ("this move was a blunder") is written as game-OUTCOME
evidence, folding into the same consensus node as real win/loss/draw results
Status: OPEN (correctness bug)
`ChessReviewIngest.cs:62-83` (`EmitBlunderOutcome`) writes a `PlyOutcome.Loss` (score 0)
attestation on the OutcomeType/OutcomeObject subject/object pair for the pre-blunder position,
purely because a blunder was detected there — using the exact same (subject, type, object)
triple that `ChessGraph.AppendMoveEdge` uses for real, game-result-derived outcome evidence.
Because `SubstrateChangeBuilder.AddAttestation` folds same-(subject,type,object) attestations
into one consensus sum irrespective of the semantic difference between "this move was
tactically bad" and "this game/position resulted in a loss," this injects synthetic "Loss"
evidence for positions that may have been objectively fine or even winning before the blunder,
independent of the game's actual result.

ISSUE C09 — opening-book ingestion systematically pulls sharp/winning theoretical lines toward
"drawish," amplified by self-play's own decisive-game overweighting — a real, traceable
consensus-poisoning risk analogous to CLAUDE.md's flagged UserPrompt concern, via a different
source
Status: OPEN
`ChessOpeningsDecomposer.cs:73-90` attests `PlyOutcome.Draw` (score 0.5) for every position on
every book line, with `games` defaulting to 4 (env-tunable 1-64) and weight 0.7. Because chess
positions are content-addressed identically regardless of source, this "book says drawish"
evidence folds into the exact same consensus node as real self-play/PGN results for that FEN,
for every source — systematically dragging genuinely sharp/winning book lines toward neutral.
Compounding: `SubstrateTurnHost.cs:99-102` gives every move edge in a NON-adjudicated (i.e.
genuinely decisive) self-play game 3x the games/witness weight of a drawn/adjudicated one
(`CheckmateGames=3`) — so whichever bias direction a given self-play engine pairing produces
(e.g. a consistently stronger White config) gets amplified 3x relative to book-line dilution,
compounding rather than offsetting it.

ISSUE C10 — UciEngine.cs: `stop` is unimplemented and architecturally unreachable; unbounded
`go depth N` is a real hang risk exercised by this repo's own tooling; malformed FEN crashes the
whole engine process
Status: OPEN (correctness bug)
`UciEngine.cs:19-58`'s command switch has no `stop` case at all (falls to a silent no-op
`default`), and `Program.cs`'s read loop is fully synchronous (blocks to completion on each
`Handle()` call before reading the next line) — so even implementing `stop` as a flag wouldn't
help without a background-thread search model; a GUI's stop can never interrupt an in-flight
search today. Separately, `ParseGo`'s `depth N` branch (`UciEngine.cs:99-100`) sets no
`MaxTimeMs` override, defaulting to `int.MaxValue`, and `Search.Think` has no external
cancellation hook (only its own internal deadline check) — and this exact code path is what
`CutechessRunner.cs:58` drives in production (`tc=inf/depth={depth}`, i.e., no time control at
all), so a deep search on a complex position can hang indefinitely with no recovery but killing
the process. Also, `SetPosition`/`Board.FromFen` throws on malformed FEN with no try/catch
anywhere in `Handle()`, so one bad `position fen ...` line kills the whole engine process. None
of the 3 are covered by `UciEngineTests.cs`'s 8 tests (test-weakness, distinct finding).

ISSUE C12 — CutechessRunner silently reports "Completed" on a run that produced zero parseable

ISSUE C13 — Zobrist hashing and the authoritative threefold-repetition check are two
independent, non-unified position-identity systems
Status: OPEN (architecture-mismatch, not a bug — both are independently correct for their scope)
`Zobrist.cs` correctly powers the search's transposition table and in-search repetition-path
avoidance (`Search.cs:99-100`). The AUTHORITATIVE game-ending threefold-repetition check
(`ChessModality.IsThreefold`, ChessModality.cs:91-99) uses a completely separate string key
(`PositionContent.Surface`) instead. Both work (verified via a passing constructed-repetition
test), but this is the same "two parallel systems doing the same conceptual job" pattern flagged
repeatedly elsewhere in this repo (01_Initial_review.txt 3.1/3.7) — a future change to one
position-identity notion (e.g. adding a rule variant) could easily update only one of the two.

ISSUE C14 — ply-cap-adjudicated self-play games are recorded as a draw for every edge, even from
decisively winning positions
Status: OPEN (data-fidelity, not a bug — a modeling simplification worth flagging)
`ModalityEngine.PlayGameAsync` (ModalityEngine.cs:203-225): `terminal ?? GameOutcome.Draw` — any
game that hits `maxPlies` without reaching a real `Terminal()` result is recorded as fully drawn
for Glicko-2 purposes for every ply, regardless of how lopsided the actual position was when
truncated. `SelfPlayEngineTests.cs:90-97` is aware of the distinction in its own assertions but
the underlying simplification stands, and would dilute/mislabel consensus signal at scale for
any self-play games that get cut off rather than played to conclusion.

ISSUE C17 — ChessCompose.Position recomputed full tier-2 composition on every call, no memoization
Status: FIXED this session (bounded in-process cache; see PART 5 for the build-time-perfcache
follow-up that would make this durable across process restarts, not yet built)
`ChessCompose.Position(string surface)` (ChessCompose.cs) recomputed the Merkle hash, centroid,
Hilbert128 encode, trajectory build, and physicality id from scratch on every call, even for a
position already composed earlier in the same run — real, avoidable cost in a corpus where
opening/early-game positions and common transpositions recur across many thousands of games.
Confirmed `SubstrateChangeBuilder.AddEntity`/`AddPhysicality` already dedupe correctly within a
batch (`_seenEntities`/`_seenPhysicalities` HashSets), so this was pure wasted CPU, not a
correctness bug — the redundant recomposition was computed then silently discarded. Fixed with a
bounded `ConcurrentDictionary<string, ChessComposed>` (cap 2,000,000 entries, unbounded would
risk real memory growth since most middle/endgame positions in a huge corpus are unique one-offs
unlike the small finite token vocabulary `TokenMemo` already caches at tier-1).

ISSUE C18 — OpeningClassifier.Classify does a full linear scan of every book line per game
Status: OPEN, not yet fixed
`OpeningClassifier.Classify` (OpeningClassifier.cs:13-37) does `foreach (var (eco, name,
lineSans) in _lines!)` — a linear scan over ALL ~3,733 loaded ECO book lines, comparing SAN
prefixes — for EVERY game ingested, called from `ChessPgnDecomposer.EmitGameOpeningMeta`. This is
O(games × book_lines × avg_line_length) with no caching/indexing, called once per game in the
entire corpus. For a mega-ingest (millions of games), this is a significant, unmemoized,
brute-force cost, worth restructuring into a trie/prefix-map over the book lines (built once at
`EnsureLoaded()` time) for O(game_length) per-game lookup instead of O(book_lines) per-game scan.

PART 5 — PROPOSED: A CHESS PERFCACHE (not yet built, documenting so it isn't lost again)
================================================================================
Confirmed via direct search this session (both a narrow `*chess*perfcache*` pattern and a broad
"perf" grep across the whole repo, including all `.claude/worktrees/*` copies) that this does
NOT exist anywhere yet — no file, no generator, no code reference. If this was discussed in an
earlier session, it was never written down anywhere this memory/scratchpad system has access to.
Recording it now precisely so that stops being true.

The concrete finding that motivates it: `ChessCompose.Position(string surface)`
(`app/Laplace.Chess.Service/ChessCompose.cs`) recomputes full tier-2 position composition
(Merkle hash, centroid, Hilbert128 encode, trajectory build, physicality id) from scratch on
EVERY call, even for a position already composed earlier in the same run. Confirmed
`SubstrateChangeBuilder.AddEntity`/`AddPhysicality` already dedupe correctly within a batch
(`_seenEntities`/`_seenPhysicalities` HashSets, SubstrateChangeBuilder.cs:52-68) — so a repeated
position's freshly-recomputed row gets silently discarded, meaning the redundant work is pure
wasted CPU, not a correctness bug. Fixed this session with a bounded in-process
`ConcurrentDictionary<string, ChessComposed>` memo (`PositionMemo`, cap 2,000,000 entries) — but
that only helps within one process's lifetime. A FRESH ingest run (the normal case — CLI
invocations are short-lived processes) starts the cache from zero again, re-deriving the exact
same ~3,733 ECO opening-book positions' full composition every single time, since those specific
positions recur across effectively every real game ingested.

Proposed real fix, matching the exact pattern already established for T0/highway
(`scripts/codegen-attestation-law.py` -> `laplace_t0_perfcache.bin` / `laplace_highway_perfcache
.bin`, both build-time-generated, mmap'd, loaded once at process start via
`CodepointPerfcache.LoadDefault()`/`HighwayPerfcache.LoadDefault()`): a build-time generator that
walks the ECO opening book (`openings/*.tsv`, the same ~3,733 lines `ChessOpeningsDecomposer`/
`OpeningClassifier` already read) and every prefix position reachable from it, computes each
one's full tier-2 composition once, and bakes the result into a `laplace_chess_perfcache.bin`
blob loaded once at process start — so every future ingest run (chess.com exports, Lumbras,
TWIC, self-play) gets the ~3,733 most commonly-recurring (opening-theory) positions pre-computed
for free instead of re-deriving them per process. Not built this session — scoped here, not
started. Would need: a new generator step (alongside `codegen-attestation-law.py`, wired into
`rebuild-all.cmd`), a binary format (can likely reuse `perfcache_format.h`'s existing
header/record conventions rather than inventing a new one), and a loader
(`ChessPerfcache.LoadDefault()` mirroring `CodepointPerfcache`/`HighwayPerfcache`'s existing
shape) that `ChessCompose.Position` checks before falling through to the in-process `PositionMemo`.
