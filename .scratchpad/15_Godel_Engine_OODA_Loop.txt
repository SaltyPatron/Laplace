================================================================================
SECTION 0 — THE THESIS
================================================================================
The walk IS the forward pass. Where a conventional transformer walks "the cat
sat on the" → "mat" through trained attention, this engine walks the same
continuation through indexed lookups over the consensus substrate — and each
step has MORE information available than a trained dot-product: the full Glicko
tuple (rating, rd, volatility, witness_count), relation-type salience, highway
bits, S³ geometry, trajectory ordinals, hilbert locality, source trust, and
provenance down to individual witnesses. Inference is C/SPI/recursion/CTE
against real indexes — microsecond steps, no GPU.

The Gödel property: the engine's own outputs are first-class inputs. A response
is content-addressed and deposited as a witness (self-reference); feedback on a
response is an attestation folding into the SAME consensus the next walk reads
(self-improvement). Evaluation IS ingestion. The loop, in OODA terms:

  OBSERVE  prompt + response witnessed into the substrate
  ORIENT   prompt_state/resolve_topic → intent routing (recall_route)
  DECIDE   walk/recall over consensus (eff_mu-ranked, indexed)
  ACT      respond; deposit the response as a witness
  FEEDBACK confirm/refute attestations → Glicko fold → the next walk differs

SECTION 1 — WHAT EXISTS TODAY (verified 2026-07-05)
================================================================================
1.1 Serving path: recall_session(prompt, session) (recall.c native intent
    router) → 12 recall_*_response SQL functions (define/describe/examples/
    languages/related/synonyms/translate/walk/what_is/relation_summary/...);
    generation = walk_text() → walk_continuations() (trajectory_generate.c:
    suffix-array n-gram descent over the corpus stream, max_order..1 backoff,
    Gumbel top-k sampling, walk_completes_floor() consensus fallback when the
    n-gram well runs dry). Beam/edge walks = generate_walk.c (batched frontier,
    ONE SPI query per level, C-side ranking).
1.2 Self-witnessing (OBSERVE/ACT): CLI chat deposits prompt (UserPromptContent,
    trust .30) and response (ResponseContent, trust .20, source
    substrate/source/Response/v1) — QueryCommands.ChatAsync. The endpoint's
    TurnWitness BackgroundService witnesses every conversation turn (bounded
    channel, drop-on-full, floor-gated). Response bigrams therefore land as
    PRECEDES attestations — the n-gram walk's OWN edges are already witnessed.
1.3 Feedback (the loop close): `laplace attest <confirm|refute> <tok...>`
    (QueryCommands.AttestAsync) — deposits PRECEDES witnesses for a token
    chain, source substrate/source/UserFeedback/v1, trust UserPrompt (.30),
    outcome Confirm/Refute, immediate consensus materialization.
    [2026-07-05: generalized — see §3 B1/B2, FeedbackContent.cs]
1.4 What the walk consumes per step: relation_rank × eff_mu(rating − 2rd) as
    the ranking key; refutation filter (rating + 2rd < neutral); witness_count
    carried for DISPLAY only (generate_walk.c). walk_continuations consumes
    corpus counts + temperature only.
1.5 NOT consumed anywhere on the read side: rd as temperature, witness_count
    as evidence mass, volatility, highway_mask gating, angular/trajectory/
    hilbert geometry, source trust classes, session/topic carry-over beyond
    single-prompt resolve. A* heuristic h=0.0 (Issue 05).

SECTION 2 — INVARIANTS (binding for this build)
================================================================================
I1  Evaluation IS ingestion. Every feedback signal, behavioral verdict, and
    self-adjudication result is an ATTESTATION from a registered source with a
    trust class — never a side table, never out-of-band state. Glicko-2 is the
    only score-keeper.
I2  Feedback binds to the TRIPLE (subject, type, object) — the same consensus
    row the walk read — never to response text. Confirming a response =
    confirming the edges that produced it.
I3  One relation vocabulary. Feedback uses existing canonical relation types;
    no new relation types without manifest (relation_types.toml) + highway
    perfcache regeneration.
I4  One deposit door. All feedback lanes go through the standard writer spine
    (SubstrateChangeBuilder → NpgsqlSubstrateWriter → ConsensusAccumulating-
    Writer). No parallel write path.
I5  One implementation per fact (Rule #6): CLI and endpoint feedback share ONE
    resolve/build/apply implementation (FeedbackContent.cs); frontends stay thin.
I6  The engine's self-signals are OUTRANKED BY DESIGN: Response trust .20,
    UserFeedback at UserPrompt .30 — one voice among many, like AiModelProbe.
    Self-confirmation cannot outshout curated sources.

SECTION 4 — LIVE LOOP PROOF (the acceptance test for the minimal loop)
================================================================================
On a seeded DB:
  1. laplace chat "<prompt>"            → response deposited (1.2)
  2. laplace attest refute <tokA> <tokB>  on a walked bigram
  3. consensus row (tokA PRECEDES tokB): rating drops / witness_count rises
     (verify: SELECT ... FROM consensus WHERE subject/type/object)
  4. re-walk the same prompt with same seed → the refuted continuation ranks
     lower where the walk consults consensus (walk_completes_floor path;
     the pure n-gram well is corpus-count-driven and unaffected — that gap is
     exactly G6a's target).
Triple form (live-verified target shape, 2026-07-05): word-level taxonomy rows do
NOT exist ('dog IS_A animal' would be a NEW claim — taxonomy lives at synset
level). A real existing row on the live DB: dog PRECEDES cart (6 witnesses,
rating 1870046278723, rd 174127722066). The triple proof targets such a row:
`laplace attest confirm dog PRECEDES cart` → witness_count+1, rating toward
confirm — verified via psql before/after, not CLI output.
CAUTION (learned live): run NO attest/chat and NO CLI rebuild while an ingest is
in flight (single-ingest law; the CLI exe also locks the build output tree).
walk_text's first call per backend builds the generation corpus
(corpus_sentence_constituents scan — minutes on a 20M-entity DB); do not run it
concurrently with a seed.
