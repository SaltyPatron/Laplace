ENGINEERING RULESET — how the substrate must be built and touched
====================================================================
Companion to 05_Substrate_Invariants.txt (which is "what must be true about the data/model").
This doc is process: how code that touches the substrate must be written, grounded in real
defects found and fixed this session, plus a full-codebase function-surface survey (2026-07-01,
264 SQL function files / ~305 total functions+views across the extension).

Rule #1: SQL/C# orchestrate; native C/C++/SPI does the heavy lifting.
    - RBAR, per-row correlated subqueries, and CTE-chained logic belong in native code with
      explicit sequenced SPI calls and bound parameters — not raw SQL switch/if trees.
    - Grounded in: define_fast/word_shape_peers_fast native rewrites of the define()/senses()/
      lexical_peers()/word_shape_peers() SQL chain — recall() went 48.7s -> ~470ms (~100x).

Rule #2: A SQL function body containing a CTE (WITH clause) can never be inlined by Postgres's
SQL-function inliner, regardless of STABLE/PARALLEL SAFE. This is a hard Postgres rule, not a
tuning issue — verified against Postgres's own inlining documentation.
    - Any function meant to be called from inside another function's hot path must avoid WITH,
      or be moved to native C.
    - Survey found 45+ CTE-bearing function files blocking inlining across the codebase —
      systemic, not isolated. Not yet fully audited/triaged by priority; see open items below.

Rule #3: Reduce result volume as early as possible in the query sequence; avoid cartesians.
    - Pre-filter on the most selective predicate before joining/computing on broad, low-
      selectivity columns. Grounded in: word_shape_peers's original cost was pre-joining on
      type_id (matches most of the vocabulary, not selective) before the KNN window narrowed
      anything — moving type_id to a post-filter on a bounded KNN result fixed it.

Rule #4: A GiST KNN `ORDER BY coord <<->> point LIMIT k` comparison point must be a genuine
bound parameter or literal.
    - A value derived from a correlated CTE reference (e.g. `me.coord` from an outer CTE) is
      NOT eligible for index-ordered KNN scanning and silently falls back to a full hash-join
      + sort — no error, no warning, just a query that's 100x slower with no visible cause.
      This was the dominant root cause of the original 48s recall() query.
    - Verify index usage via EXPLAIN; never assume a KNN-shaped query is actually using the
      KNN index just because the operator is present in the SQL text.

Rule #5: Batch, don't RBAR.
    - Compute/dedupe in bulk (client-side, or via `unnest(...) WITH ORDINALITY` array-based
      SPI batching), then write pure records once. No per-row correlated subquery re-executed
      per output row (senses_with_context.sql.in's pre-fix pattern was exactly this — fixed
      via two set-based CTEs with GROUP BY). No per-candidate SPI round trip as the default
      shape when a single array-based query will do.
    - Caveat, also grounded in measurement: batching isn't free lunch by itself — batching
      word_shape_peers_fast's per-candidate geometric checks into array queries barely moved
      timing (461ms vs 482ms) once the real bottleneck (Rule #4) was already fixed. Don't
      batch as a reflex; profile first, confirm the batch actually removes the dominant cost.

Rule #6: No duplicate/overlapping logic without a documented reason it isn't consolidated.
    - Survey found four wrappers around the same "get a definition" semantic (define(),
      define_fast(), recall_define_response(), recall_fallback_gloss()), and the
      `NOT refuted()` consensus-filter predicate reimplemented inline across 20+ functions
      instead of a shared view/helper. Every reimplementation is a place the OR-bug class
      (Rule #1 in 05_Substrate_Invariants.txt) or a future correctness fix can silently miss
      one of the copies.
    - This is the concrete first target for the "foundational set of functions/views" ask:
      a shared view layer that every consumer joins against instead of re-deriving.
      STATUS UPDATE (2026-07-05): v_consensus_unrefuted BUILT + deployed to 21 files
      (Issue 30); consensus_peer migrated too (ee3bfef); edge_rank() centralized (d061beb).
      Still outstanding: v_word_senses, the remaining NOT-refuted inliners (relate_path,
      senses_with_context x2), and the missing views from the 07 audit (v_word_points —
      ~25 inline copies of the entity+physicality type=1 join — word_anchor(),
      v_attestation_readable). Roadmap + statuses: 07 §5 re-baseline; execution order:
      .scratchpad/13 Phase 2.

Rule #7: `_fast` is the established suffix for a native-C replacement of a CTE-blocked SQL
chain (define_fast, word_shape_peers_fast). Keep using it — don't invent `_v2`/`_native`/etc.
variants that break the greppable pattern.

Rule #8: The canonical ingest pipeline is a SEQUENCE, not a set of components (author spec,
2026-07-02, stated during the OMW 20x-slowdown discussion). Having the right algorithm at the
wrong point in the sequence is a violation even if the algorithm itself is correct.
    1. Tree-sitter unpacks the packaging (raw file format) -> raw content. Nothing more.
    2. Decomposer turns raw content into substrate records. Client-side, once.
    3. Client-side dedup across the ENTIRE working set (hash-identical records merge in
       memory -- content addressing makes this a dictionary, not a query).
    4. Glicko-2 matchups accumulate CLIENT-SIDE across the set (same native glicko2.c math
       already P/Invoked; only the merge with prior DB state needs a read, and that is ONE
       bulk read of touched edge ids).
    5. ONE trunk->tier descent across the whole working set -- O(tiers) round trips per
       ingest, each a bulk set-membership probe, present nodes pruning their subtrees.
    6. The descent output is the write list. Writes are bulk raw-binary COPY of the
       client-deduped rows into per-session TEMP staging, then ONE set-based
       INSERT .. SELECT .. ON CONFLICT per table adjudicates presence AT INSERT TIME:
       DO NOTHING for content (entities/physicalities), DO UPDATE for attestations
       (present rows merge observation counts — dropping counts drops testimony).
       The client owns DEDUP (one row per id per statement — DO UPDATE faults on
       intra-statement duplicates); the server owns PRESENCE (the PK arbiter is
       always live). No standalone apply-side existence probe — the bitmap probes
       (*_stored_bitmap / *_exist_bitmap) remain compose-side (Layer A) only.
       [AMENDED 2026-07-18 — deliberate, measured reversal of the former "No
       anti-join. No ON CONFLICT. No server-side re-verification." clause: the
       apply-side probe re-verified ~12M mostly-novel ids per apply at cache-cold
       random I/O, measured 37-53 MINUTES per apply at Wiktionary scale. The DB is
       the lookup structure; adjudication rides the same PK descent the write pays
       anyway, fused with the insert.]
    STATUS RE-BASELINE (2026-07-05, verified against source — the 2026-07-02 violations
    list preserved in git history is now HISTORICAL): all five phases of the write-protocol
    plan LANDED. Working-set mode is the adapter default (one builder per record stream,
    ids probed at most once per set); apply_batch.c is DELETED (drop-files in both
    manifests) and the ingest_bulk_novel GUC no longer exists — the write lane is
    NpgsqlWorkingSetApply (pure parallel binary COPY of client-proven-novel rows,
    client-side duplicate collapse, journal-idempotent retries); Glicko accumulation AND
    the consensus fold are client-side (ConsensusAccumulatingWriter + NpgsqlConsensusFold,
    bit-parity-tested against the retired server fold). Remaining Rule #8 gaps:
    (a) step 5's O(tiers) BULK descent across the whole working set (Issue 19's
    architectural half); (b) apply of set N does not overlap compose of N+1 (apply-bound
    at chess scale); (c) seven adapter lanes feed this one spine — unification is
    .scratchpad/13 Phase 1 (Issue 45).

Rule #9: The CLI is a server workload — it runs with Server GC (DOTNET_gcServer=1 +
DOTNET_GCHeapCount=8 in env.cmd, guarded). Grounded in: parallel chess compose measured
1.9 GB/s allocation with 73 gen0 + 29 gen1 collections/s and 16% of wall time paused under
the workstation-GC default; Server GC took it to 2.4/1.6 collections/s, 6% pause, and
+70% throughput (Issue 40). Any new entry point that composes in parallel must inherit
env.cmd or set these itself.

Rule #10: Parallel pipeline stages communicate through BOUNDED channels only. An unbounded
channel between a fast producer and a slow consumer is a memory leak with extra steps
(Issue 41: fully-staged SubstrateChanges piled into RAM until the box hit 0.5 GB free).
Backpressure is the design, not a failure mode: a blocked worker during apply is correct.

Rule #11: Diagnose parallel-lane performance with independent evidence — per-thread CPU,
stack dumps, runtime counters — never with pipeline-internal rates. In a backpressured
pipeline the producer rate equals the consumer rate by construction, so "the feeder is at
X/s and compose is at X/s" identifies nothing. And a machine-wide symptom (high CPU + flat
throughput + even per-thread busyness) fits GC storms, paging, and lock convoys equally —
counters distinguish them in one 15s sample; theories don't (Issues 40-42 were found only
after three runs burned on plausible-but-wrong knob fixes).

Rule #12: Sizing defaults derive from the machine, not constants. Working-set budget is
phys/16 clamped [1,8] GiB (Issue 42 — a fixed 8 GiB OOM'd the 48 GB box while being
correct on the 128 GB server); worker counts come from CpuTopology; MKL/TBB thread counts
should come from the same topology resolution (currently a blanket 8 in env.cmd — only
laplace_synthesis links MKL, engine/core does not, verified 2026-07-03).
