## Top tier — critical & high

### 1. [CRITICAL/CONFIRMED] Tenant identity is taken from an unauthenticated client-supplied header (X-Laplace-Tenant), enabling cross-tenant data access and credit theft.
- **Where:** `app/Laplace.Endpoints.OpenAICompat/Auth/TenantResolution.cs:35`  ·  _security_  ·  src `find:endpoints-api:bugs`
- **Failure:** HeaderTenantResolver.ResolveAsync reads X-Laplace-Tenant directly from the request with no signature/token validation, and it is the ONLY tenant resolver wired (AppComposition.cs:14, LAPLACE_AUTH_MODE default 'header'). An attacker calls GET /v1/billing/usage or GET /v1/billing/entitlements with `X-Laplace-Tenant: victim-corp` and receives that tenant's full usage ledger and entitlements. Worse, POST /v1/billing/entitlements/consume with the victim's tenant header drains the victim's purchased monthly credits (EntitlementBilling TryConsumeCreditAsync scopes solely by this header). No endpoint requires any credential, so any reachable client impersonates any tenant.
- **Fix:** Require an authenticated principal (API key / bearer token mapped to a tenant) and derive TenantId from the verified credential. Only fall back to the header in an explicitly-flagged local-dev mode. Reject requests whose authenticated tenant does not match the requested tenant.

### 16. [HIGH/CONFIRMED] Unbounded re-enqueue overflows the fixed-size BFS queue in tax_bfs_up.
- **Where:** `extension/laplace_substrate/src/graph_taxonomy.c:177`  ·  _security_  ·  src `sweep-native`
- **Failure:** queue is palloc'd with exactly `cap` slots (line 92). New-node insertion is guarded by `if (n >= cap) ereport(ERROR)` at line 181, which bounds the number of distinct nodes. But the relaxation path at lines 164-177 re-enqueues an EXISTING node (`queue[tail++] = pi`) every time a shorter path to it is found, with no check that tail < cap. The number of enqueue operations is not bounded by the number of nodes: in a consensus graph where many edges yield depth improvements, the same nodes get pushed repeatedly, so `tail` grows past `cap`. queue[tail++] then writes int values past the end of the palloc'd array, corrupting adjacent heap chunks. An attacker who can shape laplace.consensus IS-A/up-type edges (or a dense ontology region) reaching a taxonomy read can trigger heap corruption and a backend crash.
- **Fix:** Guard every enqueue with a bound check, e.g. `if (tail >= cap) ereport(ERROR, ...)` before both `queue[tail++] = pi;` (line 177) and the seed/new-node enqueues, or size `queue` to an upper bound on total relaxations rather than `cap`. Better: track an `in_queue` flag per node and only enqueue when not already queued, which bounds tail by cap.
